## §1. Executive Summary

### §1.1 한 문장으로

> **"매일의 묵상이 삶의 실행으로 이어지는 도구를 만들고, 그 사업으로 형성된 자산이 전세계 작은 사역들의 지속 가능한 인프라가 되는 회사."**

좀 더 길게:

Sanctum OS는 매일의 말씀 묵상을 *영적 경험*에서 *오늘의 실행*으로 전환하는 앱이다. 묵상 → 적용 다짐 → 타임박싱 → 캘린더 연동 → 실행 체크 → AI 주간 리포트의 닫힌 루프를 통해, 사용자의 신앙이 *추상*에 머물지 않고 *구체적 삶*으로 옮겨가도록 돕는다. 장기적으로는 도트 OS라는 백엔드 인프라 위에서 묵상·가계부·의사결정·공동체 등 여러 영적·생활 영역을 통합하는 *삶 운영 시스템*으로 진화한다.

그러나 이 사업의 진짜 본업은 **자산 형성을 통한 비활성 수입**이다. Sanctum OS는 그 자산 형성의 첫 엔진이며, 형성된 자산은 *전세계 작은 사역들을 지탱하는 인프라*가 된다.

### §1.2 핵심 가설 3개

| # | 가설 | 검증 시점 |
|---|---|---|
| H1 | **묵상 → 실행 도구의 Layer 1 충성도가 형성된다** (D14·streak·자발추천 등 다중 지표) | 1차 베타 |
| H2 | **Freemium 모델에서 5% 이상 유료 전환** (정식 출시 후 12개월) | Year 1 |
| H3 | **영업이익의 40%가 자산 형성으로 전환되어 비활성 수입 엔진이 작동한다** | Year 3 |

이 셋 중 H1이 깨지면 *전체 사업 폐기 검토*. H2가 깨지면 *가격·기능 재설계*. H3가 깨지면 *Sanctum OS에 더 집중하는 방향으로 회귀*.

### §1.3 현재 상태 한 줄 진단

- ✅ **제품**: 핵심 기능 개발 완료. 베타 슬림 버전 분리 빌드 진행 중
- ✅ **누적 투자**: 30만 원 미만 (본인 인건비 제외)
- ✅ **1차 베타 명단**: 9~20명 확정 (파리·홍콩·도쿄 한인 포함)
- 🟡 **검증**: 0% (베타 시작 전)
- 🟡 **유료 사용자**: 0명
- ❌ **법인화**: 미진행 (개인사업자 등록 예정)
- ❌ **외부 자금**: 없음 (자비 운영)

### §1.4 향후 6개월 핵심 목표

1. 1차 베타 완수 → Go/Pivot/Kill 의사결정
2. Go 시 2차 오픈 베타 진입 (소그룹 단위 60~120명)
3. 정식 출시 → 첫 유료 사용자 100명 달성
4. 캐시카우 검증 — 월 매출 100만 원 돌파 (개인 생활비 보조 시작)

### §1.5 사업의 비유적 정체성 (본인이 잊지 않기 위한 메모)

> *요셉이 7년 풍년 동안 7년 흉년을 준비했듯, Sanctum OS는 풍년기에 자산 형성의 시드를 심는다. 묵상 앱 사업은 진짜 본업의 첫 엔진이고, 진짜 본업은 지속 가능한 사역을 지탱하는 자산 시스템이다.*
> 
> *사업은 사역의 도구이며, 자산은 사역의 인프라이며, 사역이 최종 목적이다.*

## §2. 문제 정의

### §2.1 진짜 문제는 무엇인가

**표면 문제 (사람들이 말하는 것)**: *"묵상을 꾸준히 못 하겠어요."*

**한 단계 깊은 문제**: 사람들은 묵상을 *하긴 한다*. 매일은 아니어도 가끔. 그런데 **묵상이 끝나면 그 내용이 어디론가 사라진다**. 책 덮으면 끝, 앱 닫으면 끝.

**진짜 문제 (이 사업의 정체성)**: **묵상은 했는데 삶이 안 바뀐다.** 영적 경험과 일상 실행 사이에 *연결 다리가 없다*. 이건 정보 문제도, 시간 문제도, 콘텐츠 부족 문제도 아니다. **인터페이스 문제**다.

### §2.2 왜 이 문제가 풀리지 않았나

#### 이유 1: 기존 묵상 앱들의 비즈니스 구조
- 묵상 앱의 수익 모델은 *"앱에서 오래 머무는 사용자"* 가 핵심
- "묵상 끝나면 빨리 앱 나가서 실행하기"는 *기존 앱 비즈니스 문법과 정반대*
- 그래서 어떤 묵상 앱도 *출구*를 만들지 않는다

#### 이유 2: 묵상 + 생산성을 둘 다 이해하는 사람이 없었다
- 묵상 앱 만드는 사람들: 신학·콘텐츠 전문가. 캘린더·타임박싱 UX 모름
- 생산성 앱 만드는 사람들: 묵상의 영적 결을 모름
- 두 세계를 동시에 이해하는 사람이 드물다

#### 이유 3: 종이 QT지 문화의 디지털 정체
- 한국 교회는 50년 가까이 *종이 QT지 문화*
- 디지털 전환이 *"종이를 화면에 옮기는 것"* 수준에 머물러 있음
- 인터랙션·실행·피드백 루프는 시도조차 안 됨

### §2.3 페르소나 4종

#### Persona A: "성실한 정체자" (가장 강한 타겟)
- 30~50대 직장인/주부
- 신앙 연차 10년+
- 매일 묵상 시도하지만 *"이거 진짜 의미가 있나"* 회의감
- 이미 종이 QT지나 한두 개 앱 사용 중
- 책·강의·수련회 등에 *돈을 써본 경험* 있음
- 핵심 통증: *"묵상은 하는데 삶이 그대로다"*

#### Persona B: "리더의 부담" (B2B 진입점)
- 소그룹 리더·청년부 간사·전도사
- 자기 묵상 + 조원들 양육의 이중 부담
- 핵심 통증: *"내 그룹의 영적 상태를 어떻게 알지?"*

#### Persona C: "디아스포라" (해외 한인 시드)
- 파리·홍콩·도쿄·LA 등 거주 한인
- 한국어 묵상 콘텐츠 갈증
- 현지 한인 교회 = 영적 + 사회적 허브
- 핵심 통증: *"한국말로, 한국 결로 묵상하고 싶다"*

#### Persona D: "고민하는 청년" (장기 타겟)
- 20~30대 초반
- 신앙은 있는데 *"왜 사는 거지?"* 단계
- 가계부 도트, 의사결정 보조 등 *삶 운영* 기능에 끌림
- 핵심 통증: *"신앙이 일상의 결정들과 어떻게 연결되는지 모르겠다"*

### §2.4 솔직한 불확실성

베타 검증 전 *100% 확신할 수 없는 부분*:
- ❓ *"묵상은 했는데 삶이 안 바뀐다"가 대다수의 통증인가, 일부의 통증인가?*
- ❓ *"묵상 + 생산성"을 자연스러워하는 사람의 비율*
- ❓ 이 통증이 *돈을 낼 만큼* 강한가

이 셋이 베타 1차의 핵심 검증 자리.

### §3.4 생태계의 6가지 진실

1. **"작은 시장에 너무 많은 플레이어"** — 한국 개신교 인구 1,000만에 25개+ 플레이어
2. **기독교 출판이 휘청거리며 디지털로 도망 중** — 두란노만 SaaS화 성공
3. **AI 신앙 시장 = "챗봇 일변도"** — ChatGPT가 더 잘함, 재방문 동기 약함
4. **교회 SaaS = "행정 자동화"에 머물러 있음** — 영적 성장 측정 없음
5. **갓피플은 "기독교 아마존" 같은 존재** — 다 묶지만 깊이는 얕음
6. **공동체 앱은 오프라인의 보조에 머물러 있음** — 디지털이 영적 공동체 인프라가 된 사례 없음

### §3.5 시장의 빈틈 (이 앱이 노리는 자리)

1. **$2.4억 투자가 "입구"에만 쏠려 있다** — 묵상 콘텐츠·오디오·UX
2. **"묵상 후 삶 적용" 영역에는 $0 투입**
3. **한국어 + 한국 교회 문화 결**의 빈자리
4. **소그룹/공동체 인프라**의 부재
5. **AI를 *영적 동반자*로 쓰는 결** — 챗봇이 아닌 동반자
6. **"실행/생산성 + 영성" 사분면 자체가 비어 있음**

### §3.6 시장 포지셔닝 맵

```
                       [영성·콘텐츠 소비]
                              │
                              │
        ●갓피플성경         ●두플러스
        ●바이블애플         ●생명의삶/큐티인
                              │
        ●YouVersion           │   ●초원 (AI 챗봇)
                              │   ●Biblely (AI 음성)
   ─────────────────────────────────────────────
   [개인]      ●갓톡           │   ★ Sanctum OS
              ●교회친구다모여   │   (묵상→실행)        [공동체]
              ●크리스천데이트   │
                              │   ●치윰 (커뮤니티+SaaS)
        ●오마이갓 (무료SaaS)  │   ●오직교적·만나·
                              │    지저스온·디모데·워칭
                              │   ●갓피플 (종합포털)
                              │
                       [행정·운영]
```

**우상단 (영성+실행) 사분면이 비어 있다.** 모든 경쟁자는 *영성-콘텐츠* 또는 *콘텐츠-생활* 자리에 있고, *영성-실행* 자리에 진입한 플레이어가 없다.

### §3.7 경쟁 우위 5가지

1. **영적 깊이 + 기술 실행력의 결합** — 10년 묵상 + 직접 개발
2. **극도로 낮은 비용 구조** — 누적 30만 원 미만, 손익분기점 매우 낮음
3. **사역 동기 → 신뢰 자산** — 교회 커뮤니티의 자연 신뢰
4. **카피 가능하나 재현 불가** — 결은 만든 사람의 영적 깊이에서 나옴
5. **단계적 해자 설계** — 시간이 갈수록 진입 장벽이 자동으로 두꺼워지는 구조

### §3.8 시장 진입의 솔직한 위험

1. **시장이 생각보다 작을 수 있다** — 적극적 묵상 인구는 더 작을 수 있음
2. **한국 교회의 디지털 도구 채택 저항** — 종이 QT지 + 카톡 의존 관성
3. **글로벌 확장의 언어·문화 장벽** — 영성의 결이 문화권마다 다름
4. **큰 자본이 카피해서 추격 가능성** — 단기 1~2년은 낮으나 3년 차에 가능

## §4. 제품 아키텍처 — 3층 구조

### §4.1 큰 그림

```
┌─────────────────────────────────────────────────────┐
│  Layer 3: 사역 인프라 (최종 목적)                     │
│   └─ 전세계 작은 사역 후원 + 선교지 지원              │
│   └─ 비활성 수익이 지탱하는 지속 가능 구조             │
├─────────────────────────────────────────────────────┤
│  Layer 2: 자산 형성 시스템 (진짜 본업)                │
│   └─ Sanctum OS 영업이익의 40% 자동 적립              │
│   └─ 금융 자산 + 부동산 + 두 번째 영리 엔진 분산       │
│   └─ 배당·임대수익이 비활성 수입                      │
├─────────────────────────────────────────────────────┤
│  Layer 1: Sanctum OS (시작 엔진)                      │
│   ├─ 표면: 묵상 → 실행 시스템                          │
│   ├─ 중간: 도트 OS 백엔드 (분화 아키텍처)              │
│   └─ 심층: 사용자 데이터 누적 = 카피 불가 해자          │
└─────────────────────────────────────────────────────┘
```

각 Layer는 *독립적으로 작동하지만 서로 연결*된다.

### §4.2 Layer 1 — Sanctum OS 제품 아키텍처

#### 4.2.1 베타 슬림 버전 (지금 빌드 중)

**핵심 루프**:
```
말씀 읽기 → 묵상 → 적용 다짐 → 타임박싱 → 
캘린더 연동 → 실행 체크 → 주간 SWAN 리포트
```

**6개 화면** (이미 스티치로 설계됨):
1. SWAN 대화형 온보딩
2. 묵상 수준 선택
3. Today (메인 묵상)
4. Application (적용 다짐 + 타임박싱)
5. Calendar 연동 확인
6. Execution 체크
7. Weekly SWAN Report
8. Floating Feedback Bubble

**스택**:
- Frontend: Web (Next.js) + Mobile (React Native or PWA)
- Backend: Firebase / Supabase (Firestore + Auth)
- AI: Gemini 2.5 Flash (저비용) → 필요 시 Claude/GPT 추가
- 캘린더: Google Calendar + iOS Calendar API

#### 4.2.2 메인 앱 (다이어트 후 정식 출시 버전)

베타 슬림 + 다음을 **단계적 추가**:
- 도트 콜렉터 (감정·사건·결정·관계·재정)
- AI 분석 리포트 (도트 패턴 발견)
- 의사결정 보조 (원칙·판례 기반)
- 워크플로우 (개인 작업 관리)
- 인물·조직 평가 (관계 도트)
- 가계부 도트 (재정 추적)

**원칙**: *베타 결과에 따라 무엇을 추가할지 결정*. 베타에서 검증 안 된 기능은 *유보*.

#### 4.2.3 도트 OS 백엔드 (Layer 1의 진짜 해자)

겉으로는 *묵상 앱*이지만 안쪽은 **세포 분화 아키텍처**.

```
도트 OS 백엔드
├─ 세포 1: SWAN (CS + 주간 리포트 작성)
├─ 세포 2: 도트 수집 엔진 (다종 데이터 수집)
├─ 세포 3: 패턴 분석 AI (도트 → 인사이트)
├─ 세포 4: 의사결정 보조 AI (원칙 + 판례)
└─ 미래 세포: 분화 대기 (가족용 / 사역자용 / 기업용 등)
```

이게 *카피 불가 본질*인 이유: 한 도메인의 데이터가 *다른 도메인의 의사결정*에 영향을 준다. 묵상 도트 + 가계부 도트 + 관계 도트가 *교차 분석*되는 순간, 외부에서 따라 만들 수 없는 *사용자 고유 인사이트*가 발생.

**전략**: 표면적으로는 *"묵상 앱이 점점 똑똑해진다"* 정도로만 노출. 도트 OS는 *물 밑*에서만 작동.

### §4.3 Layer 2 — 자산 형성 시스템

#### 4.3.1 수익 → 자산 전환 로직

```
Sanctum OS 매출
       │
       ▼
   영업이익 (세후)
       │
       ├─ 30% 생활비 + 운영
       ├─ 40% 자산 매입 ◄── 이게 핵심
       ├─ 20% 흉년 기금 적립
       └─ 10% 사역 후원 (자동 기부)
```

영업이익의 **40%**가 자산 매입으로 *자동 분류*되도록 *계좌 분리*. 본인이 *의식적으로 결정*하지 말고 *시스템적으로 강제*.

#### 4.3.2 자산 매입의 원칙 3가지

1. **확실한 것부터** — 변동성 큰 자산은 후순위
2. **현금 흐름 발생하는 것 우선** — 임대수익·배당
3. **본인 결과 맞는 것** — 영성에 거스르는 자산은 거름

### §4.4 Layer 3 — 사역 인프라

#### 4.4.1 사역 후원의 진화

```
초기 (Year 1~2):
영업이익의 10% → 즉시 후원 (작은 규모, 신뢰 형성)

중기 (Year 3~5):
영업이익 10% 후원 + 자산수익의 일부 후원

장기 (Year 10+):
자산수익이 주된 후원 재원
영업이익은 재투자에 집중 (자산 더 키우기)
```

#### 4.4.2 후원 대상의 우선순위

1. **해외 한인 교회 작은 사역** — 베타 시드와 연결 (파리·홍콩·도쿄)
2. **선교지 작은 사역** — 큰 단체 말고 현장에서 직접 사역하는 사람
3. **국내 작은 교회·소그룹** — 미디어 인프라 지원
4. **신학생·사역자 장학** — 미래 사역자 육성

#### 4.4.3 후원의 원칙 3가지

1. **익명성 유지** — 후원자 마케팅 안 함
2. **현장 지원 우선** — 본부 관리비보다 실제 사역자에게
3. **지속 가능성** — 일회성 큰 액수보다 작아도 정기적

## §5. 재무 구조

### §5.1 Freemium 가격 정책 (확정)

| 플랜 | 가격 | 포함 |
|---|---|---|
| **무료** | 0원 | 묵상 + 적용 다짐 + 캘린더 + 실행 체크 + 풍선 피드백 |
| **프리미엄** | **월 6,900원** | 무료 + AI 주간 리포트 + 도트 분석 + 데이터 무제한 + 우선 지원 |
| **연간** | 연 69,000원 (월 5,750원 효과, 17% 할인) | 프리미엄과 동일 |

#### 5.1.1 무료 vs 유료 경계의 철학

- **무료**: *습관*에 필요한 모든 것 (Layer 1 충성도 형성)
- **유료**: *깊이*에 필요한 모든 것 (자기 인식·패턴 발견)

이 경계가 *진짜 중요*. 무료 사용자가 *"이거 무료로 충분하네"* 라고 느껴야 *입소문이 폭발*. 동시에 *"더 깊이 알고 싶다"* 욕구가 생겨야 *유료로 전환*.

#### 5.1.2 가격 변경 시점 기준

- **유료 전환율 < 2%** → 가격 인하 또는 유료 기능 강화
- **유료 전환율 > 8%** → 가격 인상 검토
- **유료 사용자 이탈률 > 월 7%** → 유료 가치 제안 재설계

### §5.2 3년 매출·이익 전망

#### 5.2.1 시나리오 (Base Case)

| 지표 | Year 1 | Year 2 | Year 3 |
|---|---|---|---|
| 가입자 누적 | 15,000 | 80,000 | 300,000 |
| 유료 사용자 | 600 | 2,800 | 10,500 |
| 월 매출 | 414만 원 | 1,932만 원 | 7,245만 원 |
| 연 매출 | 4,968만 원 | 2.32억 원 | 8.69억 원 |
| 비용 | 4,500만 원 | 1.85억 원 | 4.5억 원 |
| 영업이익 | 약 500만 원 (BEP) | 약 4,700만 원 | **약 4.19억 원** |

#### 5.2.2 시나리오 분기 (Bull / Base / Bear)

```
Year 3 매출 시나리오:
├─ Bull (낙관):  15억 원  (글로벌 한인 폭발)

Bear 시나리오에서도 영업이익 약 1억 가능
→ 어떤 시나리오에서도 자산 형성 시드 확보
```

### §5.3 자산 전환의 실무 메커니즘

#### 5.3.1 계좌 분리 구조

```
[메인 운영 계좌]
   │
   ▼ 매월 말 자동 이체
   │
   ├─ 생활비 계좌 (30%)
   ├─ 자산 매입 계좌 (40%) ◄── 별도 증권사·부동산 펀드
   ├─ 흉년 기금 계좌 (20%) ◄── CMA, 즉시 인출 가능
   └─ 사역 후원 계좌 (10%) ◄── 분기별 자동 후원
```

#### 5.3.2 자동화의 철학

본인이 *의지력*으로 자산 형성하면 *언젠가 지쳐서 안 함*. 그러니까 *시스템으로 강제*. 매월 1일에 자동 이체되도록 설정하면 *의식하지 않아도* 7년 후 *자산이 쌓여 있음*.

이게 *요셉의 시스템*. 요셉이 *매일 의지로* 곡식 모은 게 아니라, *수확 즉시 저장하는 시스템*을 만든 것.

#### 5.3.3 흉년 기금의 발동 조건

```
인출 트리거:
├─ 매출이 6개월 연속 30% 하락
├─ 예상 못 한 법적·세무 비용 발생
├─ 본인 건강 위기 (수입 단절)
└─ 사업 피벗을 위한 6개월 자금 필요

비인출 원칙:
├─ "기회처럼 보이는 큰 투자" - 안 함
├─ "갑자기 좋아 보이는 새 사업" - 안 함
└─ 흉년 기금은 진짜 흉년에만
```

## §6. 실행 로드맵

### §6.1 한 페이지 타임라인

```
              Phase 0: 시드            Phase 1: 풍년      Phase 2: 다각화
              ──────────              ──────────        ──────────
Year 0  Year 1  Year 2  Year 3  Year 4  Year 5  Year 6  Year 7+
  │       │       │       │       │       │       │       │
  ▼       ▼       ▼       ▼       ▼       ▼       ▼       ▼
[베타]→[정식]→[1만명]→[유료3천]→[유료6천]→[유료1만]→[글로벌]→[자산 본격]
                                                        │
                                                        ▼
                                              [두 번째 영리 엔진 시작]
```

### §6.2 마일스톤별 결정 트리거

#### 마일스톤 1: 1차 베타 종료 (D+30)
- **측정**: 9~20명의 D14 리텐션, streak, 자발 추천
- **Go 트리거**: D14 ≥ 50%, 자발 추천 ≥ 3건, "사라지면 아쉬운가" 평균 ≥ 7점
- **Pivot 트리거**: D14 30~50%, 신호 혼재
- **Kill 트리거**: D14 < 30%, "왜 만들었는지 모르겠다" 3건+

#### 마일스톤 2: 2차 오픈 베타 종료 (D+90)
- **측정**: 60~120명 소그룹 단위, 4~6주 운영
- **Go 트리거**: 소그룹 단위 D14 ≥ 60%, 자발 *교회 도입 문의* 1건+
- **Hold 트리거**: 개인은 좋은데 공동체 효과 약함 → B2C만 정식 출시

#### 마일스톤 3: 정식 출시 후 6개월
- **측정**: 가입자 5,000명+, 유료 전환율
- **Go 트리거**: 유료 전환 ≥ 3%, 월 매출 200만 원+
- **Pivot 트리거**: 유료 전환 < 2% → 가격·기능 재설계

#### 마일스톤 4: Year 1 종료
- **측정**: 가입자 15,000명, 영업이익 BEP
- **Go 트리거**: BEP 달성 → Year 2 본격 성장
- **Adjust 트리거**: BEP 미달 → 비용 구조 점검

#### 마일스톤 5: Year 2 종료
- **측정**: 가입자 8만, 영업이익 5천만+
- **Go 트리거**: 자산 매입 시작 가능 → Phase 1 진입
- **Hold 트리거**: 이익은 있지만 *자산 매입 여력 부족* → Sanctum OS에 더 재투자

#### 마일스톤 6: Year 3 종료
- **측정**: 영업이익 3억+, 자산 누적 1억+
- **Go 트리거**: 글로벌 진입 + 두 번째 엔진 검토
- **Hold 트리거**: 한국 시장 더 깊게 파기

### §6.3 각 단계의 핵심 행동

#### Phase 0 (Year 0~2)
- ✅ 베타 1차·2차 완료
- ✅ 정식 출시
- ✅ 가입자 15,000 → 80,000
- ✅ Layer 1 충성도 형성
- ✅ 흉년 기금 12개월 운영비 적립

#### Phase 1 (Year 3~5)
- 첫 자산 매입 (안전자산 위주)
- 글로벌 한인 진입 (영어 미진입)
- 공동체 기능 본격화 (B2B 첫 교회 1~3곳)
- 영업이익 40% 자산 전환 시스템 안정화

#### Phase 2 (Year 5~10)
- 영어권 진입 (북미 한인 → 일반 영어권)
- 두 번째 영리 엔진 (도트 OS B2B SaaS or 신앙 IP)
- 부동산 본격 매입
- 자산 누적 30~50억 목표

#### Phase 3 (Year 10+)
- Sanctum OS는 *사역 도구로 진화* (수익 사업에서 영향력 사업으로)
- 자산 수익이 *주된 사역 후원 재원*
- 본인은 *시간 자유* 확보
- 후속 세대에게 *시스템 이양* 준비

## §7. 자산 포트폴리오 — 요셉 4단계

### §7.1 핵심 원칙

> *"풍년에 쓴 게 아니라 저장했다. 흉년에 자산을 매입했다. 결과적으로 생산 수단을 소유했다."*

### §7.2 단계별 포트폴리오

#### Phase 0: 시드 (~Year 2)
**목표**: 12개월 운영비 확보 + 첫 자산 매입 준비

```
구성:
├─ 현금성 자산: 80% (CMA, 단기예금)
├─ 안전자산: 15% (단기 채권, 달러 일부)
└─ 시험적 투자: 5% (소액 ETF 학습용)
```

#### Phase 1: 풍년 본격 (Year 3~5)
**목표**: 자산 누적 5~10억 + 첫 부동산 매입

```
구성:
├─ 부동산: 30% (소형 오피스텔·상가, 임대수익형)
├─ 배당주·ETF: 30% (국내+미국 우량주 분산)
├─ 현금/예금: 30% (다음 매입 대기)
└─ 금·달러: 10% (인플레 헷지)
```

#### Phase 2: 다각화 (Year 5~10)
**목표**: 자산 30~50억 + 비활성 수입 본격

```
구성:
├─ 부동산: 40% (오피스텔 다수 + 상가)
├─ 배당주·ETF: 30%
├─ 두 번째 영리 엔진: 20%
└─ 안전자산: 10%
```

#### Phase 3: 비활성 본격 (Year 10+)
**목표**: Sanctum OS 안 해도 사역 가능한 구조

```
구성:
├─ 부동산: 40% (임대 수익 안정)
├─ 두 번째 엔진 + 신규: 25%
├─ 배당주·ETF: 20%
├─ 안전자산: 10%
└─ 사역 후원 펀드: 5% (별도 운영)
```

### §7.3 자산 매입의 금지 원칙

- ❌ **빚 내서 투자** — 영업이익 범위 내에서만
- ❌ **단기 매매·트레이딩** — 장기 보유만
- ❌ **이해 못 하는 자산** — 내가 설명 못 하면 안 사
- ❌ **사역에 거스르는 자산** — 영성 안 맞으면 거름
- ❌ **유행 따라가는 투자** — 비트코인·NFT 등 FOMO 매수 금지

### §7.4 자산 운영의 학습 단계

```
Phase 0:  책 5권 + 유튜브 학습. 소액 ETF로 실험.
Phase 1:  세무사·재무 컨설턴트 1명 확보. 월 1회 자문.
Phase 2:  자산 1억 넘으면 프라이빗 뱅킹 검토.
Phase 3:  자산 10억 넘으면 전문 자산 관리 위탁 검토.
```

## §8. 리스크 & 대응 시나리오

### §8.1 리스크 매트릭스

| 리스크 | 확률 | 영향 | 대응 |
|---|---|---|---|
| 베타 실패 (D14 < 30%) | 중 | 큼 | Pivot 또는 Kill, 흉년 기금 6개월 사용 |
| 두플러스·초원이 같은 기능 추가 | 중 | 중 | 노션·슬랙 같은 *생산성/공동체 도구 방향*으로 차별화 |
| 본인 건강·번아웃 | 중 | 큼 | 흉년 기금 12개월 + 외주·자동화 강화 |
| 한국 시장 침체 | 낮 | 중 | 글로벌 분산 (Phase 1부터 시작) |
| 법적·세무 문제 | 낮 | 큼 | 초기부터 세무사 확보, 비영리 등록 검토 |
| AI 비용 폭등 | 중 | 중 | 무료 사용자 SWAN 호출 *주 1회* 제한 |
| 큰 자본의 카피 | 낮 | 큼 | 데이터 해자 + 공동체 네트워크로 대응 |
| 가족·관계 문제로 사업 중단 | 낮 | 큼 | 흉년 기금 + 자동화 시스템으로 *최소 운영* 가능 구조 |

### §8.2 블랙 스완 대응

블랙 스완 = *예측 불가능한 큰 사건*. 코로나·전쟁·기술 변화 등.

```
공통 원칙:
1. 흉년 기금 12~18개월 유지 (어떤 상황에서도)
2. 자산 분산 (부동산·금융·달러·금)
3. 플랜 B는 미리 생각하지 말고, 흉년 기금이 시간 벌어주면 그때 생각
4. 완벽한 대비는 불가능. 생존 가능한 구조만 목표.
```

### §8.3 개인사업자 → 법인 전환 체크리스트

```
다음 중 2개 이상 충족 시 법인 전환 검토:
├─ 연 순이익 5,000만 원 초과
├─ 직원 채용 필요
├─ 외부 투자 유치 의향
├─ B2B 계약 5건 이상
└─ 글로벌 진출 본격화
```

## §9. 의사결정 트리 — 상황별 액션

### §9.1 사업 운영 상황

```
[상황] 베타 1차에서 D14 < 30%
    ↓
[확인] 인터뷰에서 "왜 안 썼나" 답변 패턴
    ↓
[분기]
├─ "어려워서" → UX 재설계 + 베타 1.5차
├─ "필요 없어서" → 가설 자체 재검토 (Pivot)
└─ "다른 도구로 충분" → Kill 검토
```

```
[상황] 유료 전환율 < 2% (정식 출시 후)
    ↓
[확인] 무료 사용자 인터뷰 + 유료 기능 사용 패턴
    ↓
[분기]
├─ 유료 기능 몰라서 → 마케팅·온보딩 강화
├─ 유료 기능 가치 부족 → 기능 재설계
└─ 가격이 너무 높음 → 가격 인하 또는 무료 영역 축소
```

```
[상황] 경쟁자(초원·두플러스)가 비슷한 기능 출시
    ↓
[즉시] 패닉 안 함. 사용자 데이터 먼저 확인.
    ↓
[확인] 우리 사용자 이탈률 변화
    ↓
[분기]
├─ 이탈 없음 → 우리 사용자는 이미 우리 결에 충성. 무대응.
├─ 이탈 5~10% → 차별점 메시지 강화 + 신규 기능 1개 추가
└─ 이탈 > 10% → 카테고리 점프 (생산성/공동체 방향으로)
```

### §9.2 재무 상황

```
[상황] 영업이익 발생, 자산 매입 결정 필요
    ↓
[자동 분류]
├─ 30% 생활비 (즉시 사용)
├─ 40% 자산 매입 (별도 계좌)
├─ 20% 흉년 기금 (CMA)
└─ 10% 사역 후원 (분기 자동 이체)
    ↓
[자산 매입 계좌] 1년 누적 후 운용 시작
```

```
[상황] 매출이 3개월 연속 감소
    ↓
[즉시] 흉년 기금 건드리지 않음
    ↓
[확인] 비용 구조 점검 → 변동비 30% 절감 가능한지
    ↓
[분기]
├─ 비용 절감 가능 → 절감 후 관찰 3개월
├─ 비용 절감 불가 → 매출 회복 방안 (마케팅·기능 추가)
└─ 6개월 연속 30% 이상 감소 → 흉년 기금 발동 + 피벗 검토
```

```
[상황] 큰 투자 기회 (부동산·주식 폭락 등)
    ↓
[즉시] 흉년 기금 안 건드림
    ↓
[확인] 자산 매입 계좌 잔액으로 가능한가
    ↓
[분기]
├─ 가능 → 매입 (단, 분할 매입)
├─ 부족 → 못 사. 다음 기회 기다림.
└─ "지금 안 사면 평생 못 산다" 느낌 → 그게 FOMO 신호. 안 삼.
```

### §9.3 사역 톤 상황

```
[상황] 누가 "사역이냐 사업이냐" 물어봄
    ↓
[답변] "사업이 사역의 도구입니다. 영리 회사이지만 수익의 N%를 후원합니다."
    ↓
[원칙]
├─ 솔직함 유지 (영리임을 숨기지 않음)
├─ 사역 정체성도 숨기지 않음
└─ 둘 중 하나 과장하지 않음
```

```
[상황] 교회·목사가 "무료로 줘야 사역 아니냐" 압박
    ↓
[답변] "지속 가능한 사역을 위해 영리 구조가 필요합니다. 단, 형편이 어려운 분께는 따로 지원합니다."
    ↓
[원칙]
├─ 과소한 후원자 할인 정책 미리 준비
├─ 경제적 약자 무료 사용 옵션 마련 (소그룹 리더 신청 형)
└─ 정책으로 명확히 운영. 케이스마다 다르게 결정하면 번아웃 원인.
```

### §9.4 본인 상황

```
[상황] 번아웃 조짐 (피로·우울·일에 손이 안 감)
    ↓
[즉시] 1주일 휴식. 일 중단.
    ↓
[확인] 1주 후 다시 의욕 돌아오는가
    ↓
[분기]
├─ 돌아옴 → 일정 조정 + 자동화 강화
├─ 안 돌아옴 → 흉년 기금 사용 + 1개월 휴식
└─ 만성 → 외부 도움 (상담·외주·파트너) 적극 활용
```

```
[상황] 가족·관계 위기로 사업 손 놓아야 함
    ↓
[즉시] 흉년 기금 발동 + Sanctum OS 최소 운영 모드
    ↓
[원칙]
├─ 가족 > 사업 (예외 없음)
├─ 자동화로 6개월간 손 안 대도 굴러가는 구조 미리 구축
└─ 복귀는 준비 됐을 때만
```

### §9.5 추종자 vs 선점자 전환 시점

```
[기본 모드] 빠른 추종자 (Year 0~2)
    ↓
[전환 트리거] 다음 중 2개 이상 충족 시 선점자 모드 발동
├─ 유료 사용자 5,000명+
├─ D90 리텐션 50%+
├─ 자발 추천 비율 30%+
├─ B2B 교회 도입 5건+
└─ 영업이익으로 흉년 기금 12개월 확보 완료
    ↓
[선점자 모드] 카테고리 정의 본격화
├─ "묵상에서 시작하는 삶 운영 시스템" 메시지 전면 노출
├─ 도트 OS 백엔드 일부 공개
├─ 카테고리 리더 포지셔닝 PR
└─ B2B + 글로벌 동시 진입
```

## 🌟 마지막 한 줄

본인이 *"몰라, 그냥 해볼거임"* 이라고 했다.

이 문서가 그 *"해볼거임"* 의 *최소한의 가드레일*이다.

- 매일매일 무엇을 할지는 *모름* → 그래도 베타 시작
- 매출이 8.69억 될지는 *모름* → 그래도 Freemium 모델로 간다
- 자산 형성이 잘 될지는 *모름* → 그래도 영업이익 40% 자동 적립
- 어떤 위기가 올지는 *모름* → 그래도 흉년 기금 12개월 준비

**모르는 것을 *분석으로* 해결하지 말고, *시스템으로* 해결하기.** 그게 요셉이 한 일이다.
