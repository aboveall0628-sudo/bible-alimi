> Bridgewater 식 "Specifics up + Synthesize through time + Believability weighted" 위에
> "AI는 진단만, 처방은 절대 X, 답은 하나님 앞에서" 원칙을 얹은 구조.

## 1. 핵심 원칙 (절대 위반 금지)

1. 리포트는 **Problems + Diagnosis까지만** 담당. Design/Execute(다음에 무엇을 할지)는 사용자의 묵상 영역.
2. AI 시스템 프롬프트 고정 라인 (한국어, [reports-tone-guide.md](reports-tone-guide.md)에 정의).
3. 처방형 문장 절대 금지. 가설은 과거형·관찰형으로만.
4. 모든 리포트는 "묵상에 가져갈 질문 3~5개"로 닫음. 질문 외의 행동 제안 섹션 없음.
5. 묵상·기도 노트의 **본문(내용)은 분석에서 제외**. 메타 데이터(있음 여부·시간·소요 시간)만 사용 — 단, 카운팅·점수화는 금지 (자세한 건 §3.2 A2 재설계).
6. 인과는 단정하지 않음. "상관이 관찰됩니다"까지만 허용.
7. Bridgewater 원칙:
   - **Specifics up** — 개별 도트에서 출발해 패턴으로 합성
   - **Synthesize through time** — 시간 가로지르는 비교
   - **Believability weighted** — 반복 횟수가 많은 가설일수록 상단
8. **3층 분리** — AI 산문은 추상화된 윤곽(그림자), 사용자가 클릭하면 raw 데이터까지 노출(투명성), 그 데이터의 의미는 묵상(빛)에서.

## 2. 입력 데이터 (모든 리포트 공통)

다음 6종 데이터만 분석에 사용:

1. **도트 평가** — 실행 상태, 만족도(1~5), 결과(1~5), 원인, 라벨
2. **시계부** — 실제 시간 분배, 카테고리별 합계 (묵상 카테고리 시간이 자연히 포함되는 건 허용. §3.2 A2 참조)
3. **결단·계획** — 결단 목록, 실행 여부, 결단과 실행 사이의 시간 거리 (§3.2 A3)
4. **거래·자산·부채** — 카테고리별 지출, 활성/비활성 수입, passive ratio, giving 비율, 순자산 변화
5. **인물·조직 상호작용** — 도트 연결 인물, stance 변화, 상호작용 빈도, 인물별 도트 만족도 평균
6. **묵상·기도 메타** — 아침 게이트 수행 여부, 회고 수행 여부 (시각·소요 시간은 시계부 카테고리 합계 안에 자연히 포함되되, 별도 영적 점수화 섹션 금지)

## 3. 5계층 리포트 정의

### 3.1 일간 리포트 — Daily

- **생성 시점**: 저녁 회고 진입 시 자동 (~30초)
- **입력**: 그날의 도트·시계부·결단·거래·인물 데이터
- **모델**: `gemini-2.5-flash`
- **섹션**:
  1. 도트 카운트 (완료/부분/건너뜀/대체 분포)
  2. 계획 vs 실제 정렬도 (결단 실행률, 슬롯 겹침 비율)
  3. 시간 분배 (카테고리별 실제 시간 합계 — 묵상 카테고리 시간이 자연히 포함)
  4. 만족도·결과 분포 (평균, 표준편차)
  5. 라벨 빈도 (5축에서 가장 많이 찍힌 라벨)
  6. 인물·거래 연결 (누구와 몇 번, 어느 카테고리로 얼마)
  7. 관찰 1개 (그날 가장 두드러진 패턴, 가설 아님)
  8. 묵상에 가져갈 질문 1~2개

> **주의 — A2 재설계 반영**: "묵상 메타 수행 일수" 별도 섹션을 두지 않음. 묵상 카테고리 시간은 ③ 시간 분배의 자연스러운 일부로만 노출.

### 3.2 주간 리포트 — Weekly (토요일 저녁)

- **입력**: 일간 7개 합성
- **모델**: `gemini-2.5-flash`
- **분석 기준**: 7일 중 3일 이상 반복된 패턴만 가설로 승격
- **섹션**:
  1. 주간 도트 히트맵 (7일 × 24시간 색 분포)
  2. 요일별 패턴 (만족도 평균·표준편차·도트 수)
  3. 시간대별 패턴 (오전·오후·저녁·밤 4구간 만족도)
  4. **결단의 흐름** (A3 재설계 — 아래 참조)
  5. 라벨 상관 (동시 출현 빈도 상위 N쌍)
  6. 핀 원칙 적용 횟수와 적용 도트의 만족도
  7. 인물 카운트 (이번 주 만난 사람 빈도와 평균 만족도)
  8. 거래 합계 (카테고리별, 활성/비활성, giving 비율)
  9. **시간 분배의 결** (시계부 카테고리 합계만, 영적 정량화 X — A2 재설계)
  10. 가설 2~3개 (반복 횟수 표기 필수, 예: "3/7 일에서…")
  11. 묵상에 가져갈 질문 3개

#### A3 재설계 — "결단의 흐름" (§3.2.4)

**기존**: "빗나간 결단 공통 라벨" → 부재 강조 = 수치심 구조
**변경**: 결단과 실행 사이의 시간 거리만 추상화해서 관찰

```
이번 주, 결단에서 실행까지의 평균 거리는 2.3일이었습니다.
어떤 결단들은 시간이 흐르며 흐려지는 흐름이 관찰되었습니다.

[드릴다운] ← 사용자가 직접 클릭 시에만 raw 결단 목록 노출 (극단적 투명성)
```

**금지**: "다음 결단들이 실행되지 않았습니다: [decision_a, decision_b]"
**원칙**: AI 산문은 그림자, 사용자 클릭은 빛 — 데이터는 가리지 않되 손가락질도 하지 않음.

#### A2 재설계 — "시간 분배의 결" (§3.2.9)

**기존**: "묵상 메타 누적 (수행 일수 5/7, 평균 7분)"
**변경**: 영적 정량화 별도 섹션 제거. 묵상 시간은 시계부 카테고리 합계 안에 자연히 노출되는 것까지만 허용.

```
✅ 허용: "이번 주 묵상 카테고리 시간이 3시간 20분이었습니다" (시계부의 일부)
❌ 금지: "수행 일수 5/7", "평균 7분 12초", "지난 주 대비 +2일"
```

### 3.3 월간 리포트 — Monthly (월말 토요일)

- **입력**: 주간 4~5개 합성 + 경제·인물 데이터 본격 합류
- **모델**: `gemini-2.5-pro`
- **분석 기준**: 2주 이상 반복된 패턴만 가설로 승격. 1주짜리는 노이즈로 처리.
- **섹션**:
  1. 주간 비교 매트릭스 (4~5주의 도트·만족도·실행률 추이)
  2. 시간 4분면 (카테고리 × 만족도 매트릭스)
  3. 현금흐름 스냅샷 (active/passive, 카테고리 지출, giving)
  4. 자산·부채 변화 (월초 vs 월말 스냅샷)
  5. 인물 네트워크 (최빈 인물 상위 N명, stance 변화 카드)
  6. 라벨 상관 매트릭스 (5축 동시 출현 히트맵)
  7. 핀 원칙 효과성 (적용 도트 vs 미적용 도트 만족도 차이)
  8. **결단의 시간적 지연** (결단 → 실행 평균 일수, 흐름의 결 — A3 추상화 유지)
  9. 공동체 도트 합류 (팀별 세션 수·만족도, 가명화)
  10. 가설 3~5개 (Believability 가중 정렬: 반복 횟수 많은 순)
  11. **이번 달 자주 관찰된 패턴 N개** (A1 재설계 — 아래 참조)
  12. 묵상에 가져갈 질문 4~5개

#### A1 재설계 — "이번 달 자주 관찰된 패턴" (§3.3.11)

**기존**: "원칙 후보 1~3개" → AI가 원칙을 제안 = 처방
**변경**: 라벨에서 "원칙"이라는 처방 프레임 제거. AI는 패턴 진술만, 사용자가 원칙으로 채택할지 결정.

**출력 형식 — 도트 ID 노출 금지, 사람이 읽을 수 있는 산문으로**:

```
패턴 1 — 화요일 오후의 무거움

지난 4주 중 3주, 화요일 오후 슬롯의 만족도 평균이
다른 요일 오후보다 1.2점 낮게 관찰되었습니다.

관련된 일들:
• 4/21 (화) 14:00 — 주간 팀 회의 (만족 2, 결과 3, 라벨: 소진)
• 4/28 (화) 15:30 — 투자자 미팅 (만족 2, 결과 4, 라벨: 부담)
• 5/05 (화) 14:00 — 주간 팀 회의 (만족 1, 결과 2, 라벨: 소진·갈등)

관찰된 흐름:
- 화요일 오후 회의 도트들에서 '소진' 라벨이 반복적으로 나타났습니다
- 회의 직후 슬롯들에서 만족도가 연쇄적으로 낮아지는 흐름이 있었습니다
- 회의 시간이 평소보다 30분 이상 길어진 주에 이 흐름이 더 짙어졌습니다

[원칙 후보로 메모하기]
```

**원칙 라이브러리 라이프사이클** (라이브 시스템):
- 월간: 패턴 관찰 → 사용자가 [원칙 후보로 메모하기]로 직접 채택
- 분기: 후보의 효과성 검증 (트리거 횟수 + 적용 도트 만족도) → 정착 or archive
- 연간: 1년 검증된 원칙 → established 상태로 승격

**금지**: "이걸 원칙으로 삼으세요", "다음 달엔 ~", "추천합니다"

### 3.4 분기 리포트 — Quarterly (분기말 토요일)

- **입력**: 월간 3개 합성
- **모델**: `gemini-2.5-pro`
- **분석 기준**: 3개월 모두에서 관찰된 패턴만 가설로 승격
- **섹션**:
  1. 월간 비교 매트릭스 (3개월 핵심 지표 추이 라인)
  2. 시간 분배 변화 (카테고리별 점유율 월별 변화)
  3. 자산·부채·순자산 3개월 추이와 변화 원인 가설
  4. 인물 그래프 변화 (innerCircle 변동, stance 누적 이동, 새 카드 vs 활성 카드 비율)
  5. 원칙 효과성 누적 (분기 동안 각 원칙의 트리거 횟수와 적용 도트의 평균 만족도 — Believability weight 직접 적용)
  6. **결단 누적의 결** (3개월간 결단·실행 시간 거리의 흐름 — 라벨/ID 명시 X. A3 추상화 유지)
  7. 라벨 상관 안정성 (월간 발견 상관이 3개월 유지되는지)
  8. 7계층 목표 정렬도 (분기 목표 대비 시간 배분 실적)
  9. 가설 5~7개 (3개월 일관성 검증된 것만)
  10. **원칙 검증 결과**:
      - 월간 후보 원칙 중 효과 있는 것 → 정착 제안
      - 효과 없는 것 → archive 제안
  11. 묵상에 가져갈 질문 4~5개

### 3.5 연간 리포트 — Yearly (연말 토요일)

- **입력**: 분기 4개 합성
- **모델**: `gemini-2.5-pro`
- **분석 기준**: 4분기 모두 일관된 패턴만 원칙급 가설로 승격
- **섹션**:
  1. 4분기 비교 매트릭스 (지표 분기별 추이 + 연간 합계)
  2. 시간 배분 연간 곡선 (카테고리별 월별 점유율 12개월)
  3. 자산·부채·순자산 1년 곡선과 passive ratio 변화
  4. 인물 그래프 1년 변화 (시작 vs 끝의 innerCircle 비교, stance 누적 이동, 1년간 새 카드 vs 사라진 카드)
  5. 원칙 라이브러리 변동사 (1년간 추가·수정·archive된 원칙 전체, 각 원칙의 연간 트리거 횟수와 평균 만족도)
  6. 결단 누적 정렬도 (분기별 결단 실행률 추이의 흐름, A3 톤 유지)
  7. 라벨 5축 1년 분포 (축별 최빈 라벨 상위 N개)
  8. 7계층 목표 진척 (일·주·월·분기·연 목표별 실적)
  9. **묵상의 흐름** (1년간 묵상 카테고리 시간 누적, 영적 점수화 X — A2 재설계 유지)
  10. 공동체 활동 누적 (팀별 참여 도트 합계와 만족도)
  11. 가설 7~10개 (4분기 일관성 검증된 것만, 원칙급)
  12. **원칙 라이브러리 정착** (1년 검증된 원칙 → established 상태로 승격, 효과 없던 원칙 → archive)
  13. 묵상에 가져갈 질문 5개

## 4. AI Q&A 패턴 (A3 확장 — 모든 리포트 공통)

리포트 하단에 "리포트에 대해 질문하기" 입력창. 사용자가 자유롭게 "왜 X였어?"를 물을 수 있음.

### 출력 템플릿

```
[사용자 질문]
"왜 화요일에 만족도가 낮았어?"

[AI 응답]
관찰된 데이터의 흐름:
  ① 화요일 만족도가 다른 요일 대비 평균 1.2 낮음 (4주 중 3주)
  ② 화요일 시계부에서 회의 시간이 평균 2시간 더 길었음
  ③ 회의 도트의 만족도는 다른 도트 대비 1.2 낮음
  ④ 회의 직후 슬롯의 만족도가 연쇄적으로 낮아지는 흐름
  ⑤ 화요일 저녁 인물 P_003과의 상호작용 빈도 높음 (3/4주)
  ⑥ P_003 관련 도트의 만족도 평균이 다른 도트 대비 0.5 낮음
```

| 계층 | 트리거 |
|------|--------|
| 일간 | 저녁 회고 진입 시 클라이언트 직접 트리거 |

| 연간 | 12월 마지막 토요일 18:00 Scheduler |

### 중첩 케이스

- 토요일이 월말과 겹치면 → 월간·주간 동시 생성
- 분기말·연말 토요일은 → 모두 중첩 생성
- "월말 토요일"의 정의 = 그 달의 마지막 토요일 (월말 자체가 화요일이면 그 주 토요일은 다음 달이 됨 — 이 경우의 데이터 경계는 §10 검증 체크리스트 참조)

### 타임존

활성 타임존(별도 트랙 — `project_separate_tracks.md` 트랙 3)이 적용되기 전까지는 모든 Scheduler가 `Asia/Seoul` 기준 18:00 발화. 활성 타임존 시스템 도입 후 사용자별 `activeTz`를 따름.

## 8. UI 요구사항

- 사이드바: 📊 리포트 메뉴 클릭 시 5탭 뷰 (일간 / 주간 / 월간 / 분기 / 연간)
- 각 탭 상단에 기간 셀렉터 (날짜·주·월·분기·년)
- **두 가지 뷰 모드**:
  - 읽기 모드 (기본): 한 페이지 종합, 섹션 펼침 가능
  - 드릴다운 모드: 섹션 클릭 → 원본 도트·거래·인물 카드로 이동 (극단적 투명성)
- 리포트 화면 최하단에 시각적 구분선:
  ```
  ─────────────────────────────
  여기까지가 데이터입니다.
  다음은 하나님 앞에서.
  ─────────────────────────────
  ```
- 그 아래에 "묵상에 가져갈 질문" 카드 N개. 카드 클릭 시 다음 묵상 화면에 자동 전달.
- **AI Q&A 패턴** (§4): 질문 입력창. 답변은 항상 "내일 묵상으로" 종결. [내일 묵상 카드로 보내기] 버튼.
- 다음 묵상 화면 상단에 최근 리포트 질문 카드 자동 노출. 사용자가 답을 적으면 묵상 노트에 저장, **리포트 분석 대상에는 포함되지 않음**.
- **가설 카드에 Believability 표시**: "이번 주 3회 관찰" / "지난 3개월 일관 관찰" 등 반복 횟수.
- **A1 패턴 카드**: 산문 + 관련 일들(날짜·제목·평가·라벨) + 관찰된 흐름 + [원칙 후보로 메모하기] 버튼. 도트 ID는 사용자에게 노출 X.
- 분기·연간: 원칙 검증·정착 섹션 (월간 후보 → 분기 정착/archive → 연간 일괄 처리).
- **톤 검증 표시** (개발용 디버그 모드): 처방형 문구·영적 정량화·부재 명시 감지 시 화면에 경고.

## 10. 검증 체크리스트

### [기능]
- [ ] 일간 리포트가 저녁 회고 진입 시 30초 내 생성
- [ ] 주간·월간·분기·연간이 토요일 Scheduler로 자동 생성
- [ ] 토요일이 월말/분기말/연말과 겹치면 중첩 생성
- [ ] 5탭 뷰에서 기간 선택 및 과거 리포트 조회 가능
- [ ] 드릴다운 클릭 시 원본 도트/거래/인물 카드로 이동
- [ ] 묵상에 가져갈 질문 카드가 다음 묵상 화면에 자동 노출
- [ ] AI Q&A 패턴 동작 — 질문 입력 → 흐름 응답 → 묵상 카드 전달
- [ ] 월간 패턴 → 원칙 후보 채택 → 분기 검증 → 연간 정착 흐름 완주

### [톤·내용 — A1·A2·A3 재설계 반영]
- [ ] AI 응답에 "~하세요", "~해보세요", "추천", "내일은", "다음 주엔" 등 처방형 문구 0건 (toneValidator로 자동 검증)
- [ ] 모든 가설이 과거형·관찰형
- [ ] 인과 단정 없이 "상관 관찰됩니다" 형태만
- [ ] 각 가설에 반복 횟수 표기 (주간 "3/7", 월간 "2주+", 분기 "3개월", 연간 "4분기")
- [ ] 묵상·기도 노트의 본문이 분석에 사용되지 않음
- [ ] **A1**: 도트 ID(`dot_0421` 등) 사용자 노출 0건. 날짜·시각·제목·평가·라벨로만 표시
- [ ] **A1**: "원칙 후보 1~3개" 라벨 미사용. "이번 달 자주 관찰된 패턴 N개"로 통일
- [ ] **A1**: AI가 "이걸 원칙으로 삼으세요" 류 처방 0건
- [ ] **A2**: "수행 일수 X/7", "평균 X분", "지난 주 대비 +N일" 같은 영적 정량화 0건
- [ ] **A2**: 묵상 시간은 시계부 카테고리 합계의 일부로만 노출
- [ ] **A3**: "다음 결단들이 실행되지 않았습니다" 류 부재 명시 0건
- [ ] **A3**: AI 산문은 결단 흐름의 추상화만, raw 결단 목록은 드릴다운으로만
- [ ] **Q&A**: 모든 응답이 "이것은 '왜'의 답이 아닙니다" + "내일 묵상으로" 두 줄로 종결

### [데이터·보안]
- [ ] 가명화 후 Gemini 호출, 응답 후 역가명화 정상
- [ ] 묵상 노트 본문이 가명화 단계 이전에 입력에서 제외됨
- [ ] 민감 부분 단일 `encryptedPayload` 블롭으로 저장 (`_enc` 접미사 아님)
- [ ] Firestore Rules가 비인가 접근 차단
- [ ] AI Q&A 응답 누적 컬렉션도 동일 암호화 정책 적용

### [Bridgewater 원칙]
- [ ] Specifics up: 일→주→월→분기→연 합성 사슬 동작
- [ ] Synthesize through time: 비교 매트릭스가 모든 상위 계층에 존재

## 11. 출력 지시 (구현 순서)

1. **STEP 1** — 일간 + 주간 (회고 루프 즉시 가치)

`toneValidator`는 [reports-tone-guide.md](reports-tone-guide.md) 금지 표현 사전 기반. 위반 감지 시 AI에 재생성 요청.

## 12. 보안 — Cloud Function llmProxy 기준

- Gemini API Key는 Firebase Secrets로 관리 (`defineSecret("GEMINI_API_KEY")` — [functions/src/llmProxy.ts](../functions/src/llmProxy.ts) 참조)
- 클라이언트는 절대 Gemini를 직접 호출하지 않음. 항상 `httpsCallable(functions, 'llmProxy')` 경유
- Firebase config의 `apiKey`는 클라이언트에 노출되지만 정상 (Firebase Web SDK의 의도된 동작, Firestore Rules로 보호됨)
- 리포트 task별로 prompt 화이트리스트 (`SYSTEM_PROMPTS` 키만 허용 — 현재 dayReport / weekReport / monthReport / quarterReport / yearReport / briefing, 추가로 reportQuestion)
- 모델 화이트리스트 (`gemini-2.5-flash`, `gemini-2.5-pro`)
- 호출 인증 필수 (`req.auth` 확인)
- 토큰 예산: 연간 리포트의 분기 4개 합성이 pro 모델 컨텍스트 한계 근처 — 입력 사이즈 가드 추가 (TBD)
